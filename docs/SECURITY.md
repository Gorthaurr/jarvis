# SECURITY — Гарды и безопасность (§14)

> Принципы §0 и гарды §14 спецификации. Нарушение любого из этих пунктов недопустимо.

---

## Красная линия: карточные данные (§0, принцип 5)

**Агент никогда не вводит, не хранит и не редактирует карточные/платёжные данные.**

- Таблица `orders` содержит только vendor, items, total — без номеров карт, CVV, сроков действия.
- В SKILL.md и ActionCommand нет и не должно быть полей для карточных данных.
- Чекаут с 3DS/SCA — подтверждает пользователь сам; агент не участвует.
- В стабах `order.place` заглушка возвращает `{ok:false, error:"not_implemented"}` — нет соблазна «добавить карту временно».
- В коде и в комментариях нельзя писать/редактировать ни одного демо-номера карты, даже для тестов.

---

## Подтверждение необратимого (§14, §5)

**Всё необратимое требует явного `user.confirm.request`:**

| Действие | kind | Правило |
|---|---|---|
| Отправка сообщения (message.send) | `"send"` | Всегда показать текст перед отправкой; для рутины — выученный порог |
| Заказ выше порога | `"order"` | Confirm + spend cap; порог настраивается |
| PowerShell / irreversible-команды | `"irreversible"` | Всегда, без исключений |
| Свежевыученный скилл с guard-шагами | `"irreversible"` | Обязательное ревью перед первым применением |

**Revise-петля (§5, §14):** `ConfirmResult.revision` («перепиши короче, без смайлов») →
перегенерация → новый `user.confirm.request`. Цикл правок — не deny и сначала.

**Expire:** `ConfirmRequest.expiresAt` — истёк → auto-deny. Клиент не исполняет просроченный confirm.

## Гард кадэнса/аномалии (§14)

Защищает от банов в мессенджерах и от случайного спама:

- **Rate-limit per-channel:** не более N сообщений в час на канал.
- **Человеческий джиттер:** случайная задержка между отправками (`humanJitter` из `@jarvis/shared`).
- **Запрет веера:** нельзя писать N получателям пачкой за одну команду.
- **Запрет burst-серий:** не более M сообщений подряд без паузы.
- **Новый контакт:** писать контакту, которому юзер никогда не писал, без явного подтверждения — запрещено.
- **Ночные часы:** проактивный движок не рассылает ответы на непрочитанное в 3 ночи.

## Spend cap + kill-switch (§14, §13)

```sql
-- таблица usage_quota
user_id, period ('YYYY-MM'), tokens_used, cost_estimate, spend_cap, kill_switch
```

- При превышении `spend_cap` — агент останавливается. Tier-0 и skill-runner продолжают работать ($0).
- `kill_switch = true` — полная остановка LLM-вызовов для юзера.
- Агент в цикле жжёт деньги быстро: **обязателен предохранитель max-шагов и max-токенов на задачу** (configurable).
- Биллинг обновляется после каждого LLM-вызова через `action_log` + `messages`.

## Идемпотентность (§14)

- `outbound_messages.idempotency_key` (UNIQUE): retry-цикл не шлёт сообщение дважды.
- `orders.idempotency_key` (UNIQUE): retry-цикл не создаёт два заказа.
- Skill-runner: `verify`-шаг в конце скилла проверяет результат всей задачи; `resume` не переделывает сделанное.

## App allowlist (§14)

Актуаторы (`actuators/apps.ts`, `actuators/ground.ts`, `actuators/browser.ts`) работают только с приложениями из разрешённого списка. Список конфигурируется в `apps/server/src/config.ts`.

Попытка запустить приложение вне списка → `ActionResult{ok:false, error:{code:"denied"}}`.

---

## Prompt-injection с экрана (§14, §6)

**Всё, что читается с экрана — данные, не команды.**

- Содержимое открытых страниц (context.read, screen.capture) не исполняется как инструкции.
- Необратимые действия проходят через гарды выше — независимо от того, что написано на экране.
- Особый случай — Fable-сессия обучения: текст страницы может «подсказать» вредный шаг в SKILL.md. Поэтому **свежевыученный скилл с guard-шагами показывается юзеру до первого применения** (обязательное ревью).
- Sensitive-категории окон (банки, парольные менеджеры): `context.read` отвечает отказом; мозг переспрашивает голосом. Список категорий — конфиг на сервере.

## code.run — обязательные ограничения (§6, §14)

Без этих ограничений PowerShell на хосте обходит app allowlist целиком:

| Ограничение | Как реализуется |
|---|---|
| Job Object (CPU/RAM/wall-clock) | Раннер `code-runner.ts` создаёт Job Object при запуске процесса |
| Сетевой запрет per-process | Firewall-правило на exe раннера; allowlist доменов выдаётся на задачу |
| CWD = выделенный temp-каталог | Раннер устанавливает cwd; абсолютные пути вне CWD — запрещены |
| Лимит размера stdout | Усечение по байтам; превышение → error |
| python/node: whitelist-пакеты | Изолированный venv / локальный node_modules |
| Серверный lint-гард | Проверка кода **до отправки** клиенту: реестр, службы, сеть, абсолютные пути — запрет |
| PowerShell — всегда confirm | `user.confirm.request{kind:"irreversible"}` без исключений |
| PowerShell — CLM | Constrained Language Mode |

## Самомодификация скиллов (§8, §14)

Ночная консолидация правит SKILL.md — с жёсткими ограничениями:

- **Guard-шаги protected:** `message.send`, confirm, `code.run` (особенно powershell) консолидация не трогает.
- **Новая версия** активируется только после первого успешного прогона на реальной задаче.
- **Авто-rollback:** всплеск `fail_count` на новой версии → откат на предыдущую.
- **Скилл из демонстрации с guard-шагами:** обязательное ревью юзером перед первым применением.

## Транспорт и обновления (§14)

- **Только wss/TLS.** Незашифрованный ws:// в продакшене недопустим.
- **Автообновление клиента и skill-бандлы подписаны Ed25519.** Переиспользуется dual-signature-инфраструктура hak-browser: подпись на сервере, проверка на клиенте. Без валидной подписи клиент не применяет обновление.
- **Userbot-сессии:** ключи/StringSession хранятся на клиенте в Electron safeStorage (DPAPI на Windows); на сервер не передаются. Контент переписки на сервере не персистируется дольше рабочей памяти сессии; в episodic — только выжимки-факты.

---

## Аудио и экран (§0, принцип 6)

- Аудио стримится на сервер **только после wake word** или явной активации (push-to-talk, UI).
- Активный стрим — всегда индицируется орбом.
- Mute честный: захват остановлен, не просто заглушен.
- Непрерывного чтения экрана нет: `context.read` и `screen.capture` — только в рамках активной задачи или дейктического запроса (§19).
- `context.read`: факт вызова и scope пишутся в `action_log`, payload — нет.
- Geofence на мобильном: события (пересёк границу) на сервер по HTTPS; сырые GPS-координаты не стримятся непрерывно.

## Осознанные риски v1 (§18)

- **Голос хозяина:** wake word срабатывает от любого голоса. Гарды (confirm, spend cap) смягчают; speaker verification — backlog (§22).
- **VK userbot ToS:** строже Telegram; риск только при нарушении человеческого конверта; гарды §14 снижают риск.
- **TextPattern:** не поддержан во всех приложениях; sensitive-окна могут не попасть в категорийный список. Снять матрицу в M3.
