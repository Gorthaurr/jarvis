# Джарвис как СРЕДА ИСПОЛНЕНИЯ: концепт-документ

> Как построить для голосового мажордома собственную среду, в которой облачный Claude делает на твоём ПК почти всё — и делает это за ОДИН раунд-процедуру, а не за серию облачных кликов.
> Синтез 8 измерений исследования (2025-26) поверх существующего стека Джарвиса.

---

## 1. TL;DR

**Да, «среда, где модель делает всё быстро» — реалистична, и это не гипотеза, а сходящийся индустриальный паттерн 2025-26.** Ключевой архитектурный выбор один: **перестать эмитить действия по одному tool-call JSON и начать эмитить ПРОЦЕДУРУ кодом**, исполняемую в живом интерпретаторе на твоём ПК, где наблюдение (есть ли кнопка, появился ли текст) — дешёвый локальный вызов внутри кода, а не отдельный облачный раунд.

**Где главный скачок.** Не в смене модели — Claude уже умеет GUI на уровне человека (OSWorld к 2026: Claude Sonnet 4.6 = 72.5%, human baseline = 72.36%). Скачок — в СРЕДЕ, и он измерен на порядки:
- CodeAct: −30% раундов, +20 п.п. success (код-как-действие против JSON tool-call).
- Anthropic programmatic tool calling: −37% токенов, ~19 устранённых inference-раундов при 20+ вызовах в одном коде.
- ActionEngine (программный синтез процедуры + state-machine-память интерфейса): **−94% LLM-вызовов, 2× быстрее, 11.8× дешевле, 95% vs 66% success** — это самый радикальный ориентир под цель «раунд = процедура».
- UFO2 (Microsoft, тот же Windows/UIA-стек, что у Джарвиса): −51.5% инференса спекулятивным батчем с precondition-гардом.
- Agentic Plan Caching (= то, что делает HERMES): −50% стоимости, −27% латентности при 96.6% качества.

**Где стена (честно).** Среда схлопывает СЕРИЮ раундов, но:
- генерация самого кода — по-прежнему один полный ход Claude (2-13с): выигрыш в устранении последующих ходов, не первого;
- цифры −30..−94% достигаются ТОЛЬКО когда наблюдение дёшево внутри кода → значит world-model и событийная подписка — не «ещё одна фича», а обязательный API внутри SDK, иначе код кликает вслепую;
- server-side world-model системно ВРЁТ при рассинхроне (устаревшее дерево → неверные действия с частотой 0.91-1.00) — её нельзя доверять на необратимом без пиксельной сверки;
- событийное наблюдение (UIA-events) и живая server-world-model в литературе продакшн-систем почти НЕ описаны — это фронтир, который Джарвису придётся строить самому, а не копировать.

**Вывод.** Джарвис архитектурно ближе всего к UFO2 и уже независимо реализовал 2 из 3 её приёмов (input_batch + SkillStep.precondition). Максимальный неиспользованный рычаг — не у Windows-агентов, а у **ActionEngine + CaveAgent**: persistent-SDK с состоянием + компиляция процедур. Первый шаг — превратить одноразовый `code_run` в **живой stateful-интерпретатор с фасадом `jarvis.*`** поверх уже существующего сайдкара/расширения.

## 2. Диагноз: почему управление ПК сейчас медленное и дорогое

**Корень — арифметика `раунды × облачный ход`, где узкое место — облако, а не актуатор.** Прямое эмпирическое подтверждение (OSWorld-Human, июнь 2025, анализ Agent S2 на OSWorld):
- **75-94% всей латентности задачи** уходит на LLM-вызовы (планирование + рефлексия), НЕ на исполнение действия;
- агенты делают в **1.4-2.7× больше шагов**, чем человек;
- каждый следующий шаг до **3× дольше** начального (растёт история контекста);
- задача, которую человек делает за 2 минуты, у агента занимает >20 минут;
- планирование съедает 53% времени, рефлексия ещё 33%.

То есть каждый облачный ход Claude (у Джарвиса это 2-13с генерации) — главный враг. Наивный baseline — это Anthropic computer-use screenshot-loop: клик→скриншот→клик, 2-5с/шаг, каждый скриншот = ~1.15 Мпикс vision-токенов, нет accessibility-дерева, нет батчинга, нет памяти процедур. Строить среду на нём — шаг назад.

**Что Волны 2-3 УЖЕ схлопнули** (Джарвис здесь впереди голого computer-use):
- **Fused act+observe** (2.1): актуатор сам прикладывает дешёвое наблюдение (a11y-выжимка / OCR) в тот же tool_result → verify-долг снимается в том же раунде, паттерн «клик→скрин→клик→скрин» схлопнут (−35-50% раундов GUI-задач).
- **input_batch** (2.2): до 12 механических шагов одним вызовом под одной арендой; независимые ЧИТАЮЩИЕ вызовы диспатчатся параллельно.
- **Дешёвые сенсоры** (2.3): `screen_read_text` (локальный WinRT OCR в сайдкаре), `wait_for` (клиентский поллинг без LLM-раундов), `screen_probe` (перцептивный хеш).
- **ui_snapshot** (2.4): set-of-marks — интерактивные элементы окна {handle,role,name,bbox} вместо 2K-скрина, ui_invoke по handle.
- **Precondition + executor-тир + реплей прежде петли** (Волна 3): проверка живого UIA-стейта перед шагом; откат на дешёвый тир при известной процедуре; REPLAY_SAFE-навыки исполняются ДО петли.

**Что осталось узким местом (несмотря на Волны):**
1. **Каждое действие — всё ещё отдельный tool-call JSON и отдельный (или тесно связанный) облачный раунд.** input_batch — только «механика», needsLlm-шаги в него запрещены. Композиция (циклы/условия/ветвление по наблюдению) требует нового хода модели.
2. **Наблюдение — по-прежнему опрос, а не подписка.** `wait_for`/`screen_probe` поллят; UIA-события ОС не используются вовсе. «Разбуди, когда появится X» стоит поллинга.
3. **Нет живого состояния между ходами.** `code_run` спавнит процесс НА КАЖДЫЙ вызов (stateless) → HWND окон, COM-объекты, UIA-элементы не переживают вызов; модель переизобретает состояние каждый ход.
4. **Grounding идёт через облачный раунд.** Модель смотрит `ui_snapshot` и решает — это дорогой ход, а не локальный `$0`-запрос к кэшу дерева.
5. **HERMES-навыки реплеятся как последовательность actions**, а не вызываются как именованные функции — recall семантический («инжект похожего markdown»), модель НЕ пишет вызов по имени.

## 3. Что показал рынок и наука 2025-26 — по каждому столпу

### 3.1 Code-mode: раунд = процедура
- **CodeAct** (ICML 2024, arXiv 2402.01030): код-как-действие даёт **+20 п.п. success и ~30% меньше шагов/токенов** против JSON/text tool-call. На M3ToolEval (82 многошаговых задачи) GPT-4 = 74.4% (CodeAct) против 52.4% (JSON); среднее число ходов 5.5 vs 7.6. Причина: код нативно умеет циклы/условия/композицию — то, что в JSON требует раунда на шаг. Воспроизведено независимо: **HuggingFace smolagents CodeAgent** (~30% меньше шагов и LLM-вызовов).
- **Anthropic «Code execution with MCP»** (ноя 2025): переписав workflow с прямых tool-call на код, токены **150 000 → 2 000 (−98.7%)**. Инструменты подаются как файловая система TS-модулей, агент читает определения on-demand (progressive discovery), фильтрует данные В среде до попадания в контекст. Продакшн-валидация: GitHub MCP, 112 инструментов, те же ~98%.
- **Anthropic «Programmatic tool calling»**: **−37% токенов** (43 588 → 27 297) и ~19 устранённых раундов при оркестрации 20+ вызовов в одном код-блоке; точность РАСТЁТ (GAIA 46.5%→51.2%, knowledge retrieval 25.6%→28.5%). API сам конвертирует tool-definitions в Python-функции.
- **Cloudflare Code Mode**: один инструмент `codemode({code})`, MCP-схемы автоконвертируются в типизированный TS-API; конвертация режет токены **на 81%**. Ключевой инсайт: LLM видели МНОГО кода и почти не видели «tool calls» (спец-токены без природных примеров → ненадёжность). Наружу — только через авторизованные bindings (нет утечки ключей). Это **ровно capability-идея Джарвиса**: `jarvis.window('Dota')` = уже авторизованный binding, а не сырой SendInput.
- **Persistent-state поддержан провайдером**: `code_execution_20260120` сохраняет состояние Python-интерпретатора в рамках разговора, контейнер переиспользуется по `container_id`. НО: state НЕ делится между разными средами → живой процесс-интерпретатор надо держать **на клиенте (Windows-ПК)**, а не в облачном контейнере Anthropic, чтобы драйвить локальную ОС.

### 3.2 Persistent stateful REPL
- **CaveAgent** (arXiv 2601.01569, 2026) — прямой прототип концепта (в): dual-stream, LLM генерирует Python, состояние живёт в персистентном IPython-kernel; инструменты/данные инжектятся как ЖИВЫЕ объекты в namespace. Цифры: **+10.5% success (Tau2-bench retail), −28.4% всех токенов, −32.7% prompt-токенов за счёт меньшего числа раундов, −59% на data-intensive**. Безопасность — AST-валидация перед exec.
- **Критическая стена — snapshot-персистентность не годится.** LangChain deepagents: «Snapshots preserve only serializable data. Functions, classes, unserializable objects become inaccessible». HWND/COM/UIA/WS-хендлы НЕсериализуемы → нужен НАСТОЯЩИЙ живой процесс (state в RAM), не replay/snapshot.
- **Персистентность = поведенческий приор.** arXiv 2603.01209: модель, не выровненная с runtime, платит «amnesia tax» — **×3.5 токенов** (18.6K vs 67.9K), переизобретая состояние, ИЛИ даёт reference-errors на несуществующие переменные (80% эпизодов). Claude НЕ дообучен под нашу SDK → **модель состояния надо ЖЁСТКО документировать в персоне**: что живёт между ходами, что переживает рестарт, `print()` для сверки.
- **PowerShell Runspace / InitialSessionState** — нативный long-lived stateful исполнитель на Windows: переменные/модули/COM живут «between open and close runspace»; constrained runspace = capability-модель. Хостится в **уже существующем C#-сайдкаре** — без внешних зависимостей.
- Экосистема sandbox (E2B/Blaxel/Modal) даёт lifecycle-рецепт (warm standby, resume <25мс, kernel по UUID+PID, readiness-wait), но их ИЗОЛЯЦИЯ Джарвису вредна — брать только форму управления жизненным циклом (у Джарвиса уже есть в SidecarClient с авто-рестартом/backoff).

### 3.3 Windows GUI-агенты
- **UFO2** (arXiv 2504.14603, Microsoft, апр 2025) — прямой прототип: HostAgent (планировщик) + AppAgent (исполнитель per-app), grounding-гибрид UIA + OmniParser-v2 (дедуп по IoU>10%). **Speculative multi-action**: один запрос предсказывает несколько действий, каждое валидируется precondition по UIA перед исполнением → **−51.5% инференса, −58.5% шагов на o1**. Джарвис уже реализует ~60% этой архитектуры (input_batch + precondition). Cross-app остаётся хрупким у ВСЕХ (UFO2 = 9.1%).
- **ActionEngine** (arXiv 2602.20502, 2026) — самый радикальный под концепт владельца. Офлайн строит State-Machine Graph (узлы=состояния, рёбра=операции; статика отделена от данных → ~20-30 состояний/сайт). Онлайн: за ОДИН вызов выдаёт «sketch-программу», linking резолвит плейсхолдеры в пути графа (BFS), компиляция → Python. **−94% LLM-вызовов, 118с vs 237с, $0.06 vs $0.71, 95% vs 66% success.** Самолечение: Python-ошибка → hot-patch, UI-провал → vision re-grounding.
- **Agent-S2** (arXiv 2504.00906) — Mixture-of-Grounding (Visual UI-TARS / Textual OCR / Structural cells) + проактивный реплан после КАЖДОЙ подзадачи (+4.6-6.1 п.п.). OSWorld 34.5%.
- **OSWorld 2026**: Claude Sonnet 4.6 = 72.5%, GPT-5.4 = 75.0%, human = 72.36% (ранний baseline был 12.24%). **Стена сместилась с «модель не понимает GUI» на «архитектура/скорость/стоимость раундов».**
- Консенсус: **hybrid API+GUI** (UFO2 Puppeteer: Excel→xlwings, Outlook→win32com, GUI фолбэком). Обзор arXiv 2503.11069 «API vs GUI Agents».

### 3.4 World-model / OmniParser / декларативный OS-интерфейс
- Запрос к структуре **~100 мс против 2-5с** на скрин→VLM; DOM-native исполнение срезало латентность действия до **<500 мс** и токены ~10× (OSWorld-Human, InfoWorld).
- **Стена world-model**: генерация полного a11y-дерева = **3-26с** на сложное приложение. `<100 мс` — это ЧТЕНИЕ уже построенного дерева. Наивный ре-скан UIA каждый раунд НЕ дешевле скрина → экономику даёт ТОЛЬКО инкремент.
- **Рассинхрон системно врёт** (arXiv 2607.04334 «Do GUI Agents Believe Their Eyes?»): 37.4% экранов имеют ghost-узлы; при устаревших ссылках модель «видит» отсутствующий элемент в 0.57-1.00 случаев; под конфликтом пиксели↔структура — ошибочные действия 0.91-1.00 vs ≤0.05 при согласии. Иконки pixel-bound (0.80-0.98) → чисто структурная модель принципиально неполна.
- **Декларативный OS-интерфейс** (EuroSys '26, arXiv 2510.04607): вместо скриншот+координаты — декларативное описание доступного + recognition-over-recall. Прямо ссылается на Windows UIA / Control Patterns.
- **OmniParser v2** (Microsoft, фев 2025): YOLOv8 + Florence-2, −60% латентности (~0.6с/кадр), закрывает UIA-слепые окна (игры/canvas) — на RTX 5080 локально реально.
- Set-of-Marks «does not incur significant overhead» и сокращает шаги vs полное a11y-дерево — `ui_snapshot` Джарвиса уже на правильном представлении.

### 3.5 Событийная подписка UIA
- `IUIAutomation` даёт 5 методов подписки: `AddAutomationEventHandler` (вкл. `UIA_Window_WindowOpenedEventId`), `AddFocusChangedEventHandler`, `AddPropertyChangedEventHandler`, `AddStructureChangedEventHandler`, `AddNotificationEventHandler`. Область — `TreeScope`. Callback = 0 токенов, 0 раундов, мгновенно. **Прямая замена wait_for-поллингу.**
- **Жёсткие правила** (MS Learn): callback-и на отдельном non-UI потоке в **COM MTA** (STA ломает удаление хендлеров); callback ДОЛЖЕН возвращаться <100 мс, иначе Windows тормозит весь GUI и сам отпишет handler → в callback только enqueue; **НИКОГДА не подписываться Subtree/Descendants на корень рабочего стола** (system-wide локи вешают машину); событие может прийти ПОСЛЕ отписки → access violation (держать сильную ссылку, Remove на том же потоке).
- **События НЕ полны**: Win32/WinForms-прокси и кастомный canvas/WebGL (Dota, игры) не поднимают все property-changed → нужен ГИБРИД event + редкий reconcile-поллинг.
- `SetWinEventHook` (WINEVENT_OUTOFCONTEXT) — дешёвый слой грубых оконных событий (FOREGROUND/SHOW/FOCUS), отдаёт HWND/objid → связка «WinEvent триггер → UIA cached snapshot».
- На событии обновлять модель через **CacheRequest (bulk fetch)** — весь набор свойств поддерева за один cross-process вызов вместо чтения по свойству.

### 3.6 Компиляция навыков (Voyager и далее)
- **Voyager** (arXiv 2305.16291): навык = исполняемая функция, сохраняется ТОЛЬКО после self-verification (по состоянию мира, не по «ok»), индексируется эмбеддингом описания; новые навыки вызывают старые (temporal composition).
- **AWM** (ICML 2025): индукция workflow из траектории через абстракцию конкретики в слоты («dry cat food»→«{product-name}»). WebArena +51.1% относит. (35.5% vs 23.5%). = это `skill-slots {{slot}}` Джарвиса.
- **ToolMaker** (ACL 2025): навык → ТИПИЗИРОВАННАЯ функция с сигнатурой, unit-test-гейт + closed-loop self-correction (80% из 15 задач). Роль теста приёмки играет `wait_for`/`ui_snapshot`.
- **SkillOps** (2026): навык = контракт (Preconditions, Operation, Artifacts, **Validators**, Failure-modes); библиотека = граф с dependency/redundancy/alternative; body-hash дедуп; обслуживание ≤9 LLM-вызовов на 2000 навыков (~$0.0026). ALFWorld 200 навыков: 79.5% (+8.8 п.п.), держит ~80% при 200→2000.
- **Agentic Plan Caching** (arXiv 2506.14852): **−50.3% стоимости, −27.3% латентности, 96.6% качества** — академическая формулировка HERMES-реплея.
- **ГЛАВНАЯ СТЕНА — «More Skills, Worse Agents» (2026)**: рост библиотеки САМ ухудшает агента (skill shadowing / negative transfer). LLM-авторские навыки **+0.0 п.п.**, человеко-курированные **+16.2 п.п.** Вывод: компилировать ТОЛЬКО из верифицированных человеко-показанных демо, держать ЕДИНСТВЕННОГО канонического представителя на capability, retire провальных. У Джарвиса уже есть 3 редких предохранителя (семантич. дедуп 0.9 + fail_count-подавление + гард полярности) — сильная стартовая позиция. ADAS-автономию (сам сочиняет из воздуха) НЕ брать — дорого, не окупается (arXiv 2510.06711).

### 3.7 Capability-безопасность
- **Claude Agent SDK**: 6-ступенчатый гейт, единственный универсальный барьер — **PreToolUse-hook** (раньше всех, deny держит даже в bypass); allow-правила обходят позднюю логику; `requiresUserInteraction`-инструменты ВСЕГДА падают в confirm.
- **OpenAI Operator/CUA**: три режима вместо одного confirm — confirmation (перед необратимым внешним эффектом), watch mode (почта — активный надзор), takeover mode (креды вводит пользователь, агент не скриншотит). Отказ от класса задач (банк).
- **Meta «Rule of Two»**: в одной операции максимум 2 из 3 свойств — (A) недоверенный ввод, (B) чувствительные данные, (C) необратимое действие; все три = RCE-класс, обязателен человеческий гейт. «Суммаризация письма не должна триггерить shell, что бы ни говорило письмо».
- **Информационно-поточный контроль** (CaMeL/FIDES/MELON): недоверенный контент читает изолированный LLM без tool-calling, вызовы гейтятся по провенансу. AgentDojo: CaMeL ASR 23%, MELON-Aug 0.32%, FIDES 0% внутри. Anthropic Opus 4.5 адверс-тренингом ~1% ASR — «но 1% всё ещё значим».
- **Lies-in-the-loop**: инъекция манипулирует ТЕКСТОМ диалога подтверждения → человек одобряет перефразировку. Защита: показывать СЫРУЮ операцию (детерминированные поля ActionCommand), не NL-резюме модели.
- **Prompt-injection в GUI НЕ решается на уровне модели**: «Attacker Moves Second» — 12 защит обойдены >90% адаптивно, ред-тимеры 100%. Работает только детерминированное: egress-allowlist по доменам, стрип Unicode Tag U+E0000-E007F, мульти-канальная классификация доверия, default-deny на необратимое.
- **Memory poisoning** (Unit 42 на Bedrock): вредоносные инструкции в памяти переживают сессии → у Джарвиса durable HERMES/consolidation = вектор. `looksLikeDirective`-фильтр + untrusted-обёртка уже есть — покрытие надо аудитить.

### 3.8 Экономика раундов — синтез
Отдельные приёмы дают десятки %, вместе — кратно. API-first −58.5% шагов; батч с UIA −51.5% инференса; code-mode −30% ходов; plan cache −50% стоимости; спекуляция 2-5× латентности (PASTE −48.5%). Ключевой инсайт: **раунд дорог из-за облачного вызова, не актуатора → любой приём, убирающий РАУНД, выигрывает непропорционально.**

## 4. Целевая архитектура среды Джарвиса

Четыре столпа связываются в один стек поверх существующего кода. **Не альтернативы — один конвейер.**

                │                                  │
┌───────────────▼──────────────────────────────────┴───────────────────┐
│  КЛИЕНТ + C#-САЙДКАР (Windows-ПК)                                     │
│  • PERSISTENT-ИНТЕРПРЕТАТОР: живой процесс со СТАНИЕМ между ходами.    │
│    - PowerShell Runspace (long-lived) в C#-сайдкаре для PS/UIA-ветки. │

**Где что живёт:**
- **World-model — на СЕРВЕРЕ** (кэш дерева, поколения, инвалидация), потому что именно сервер собирает промпт и решает, дёшев ли запрос «есть кнопка». Но ИСТОЧНИК истины — сайдкар, стримящий дельты.
- **Persistent-интерпретатор — на КЛИЕНТЕ/сайдкаре**, потому что state (HWND/COM) несериализуем и должен драйвить локальную ОС; облачный контейнер Anthropic для этого не годится.
- **Событийный слушатель — в C#-сайдкаре** (у него уже COM/UIA-доступ).

**Что ПЕРЕИСПОЛЬЗУЕТСЯ (Джарвис ~60% пути):**
- `ui_snapshot` set-of-marks + реестр хендлов + `ui_invoke` — готовое правильное представление узлов world-model;
- C#-сайдкар (UiaGrounder) — хост для long-lived Runspace И событийного слушателя;
- `code_run` (python/node/powershell, wall-clock timeout) — фундамент интерпретатора;
- HERMES (recall/replay/slots/precondition/fail_count/полярность) — кандидаты в SDK-функции + 3 предохранителя shadowing;
- `COLD_TOOL_NAMES`/`tool_load` — готовый progressive-discovery для каталога SDK-функций;
- `wait_for`/`watch`-предикаты/`gsi-listener` — резолвятся событиями вместо поллинга;
- `SidecarClient` авто-рестарт/backoff — lifecycle-рецепт для kernel;
- `untrusted_content`/`self-guard`/`SSRF-гард`/`requestConfirm`/`looksLikeDirective` — база capability-слоя;
- fused observe (observe.ts) — пиксельная сверка перед необратимым (митигация рассинхрона).

**Что НОВОЕ:**
1. Персистентный интерпретатор с живым namespace (сейчас stateless spawn).
2. SDK-фасад `jarvis.*` поверх актуаторов (сейчас каждый — сырой ActionCommand).
3. Серверный кэш дерева с поколениями + событийный push-канал сайдкар→сервер (сейчас в основном сервер→сайдкар).
4. Компиляция HERMES-процедур в вызываемые именованные функции + обязательный Validator-контракт.
5. OmniParser v2 локально для UIA-слепых окон.
6. Rule-of-Two taint-трекинг (untrusted→irreversible гейт).

## 5. Столп «раунд = процедура» — детально (главный скачок)

### Как это работает
Вместо серии `input_click` → `screen_capture` → `input_click` (каждый = облачный ход), Claude пишет ОДИН код на jarvis-SDK, исполняемый в persistent-интерпретаторе за ОДИН раунд, где наблюдение встроено:

```python
# ОДИН ход Claude вместо 8-15:
w = jarvis.window("Dota 2")
w.focus()
jarvis.wait_for(text="Найти игру", timeout=10)   # локально, $0, без LLM
w.click(role="button", name="Найти игру")
# ждём появления кнопки Принять — событийная подписка, не поллинг:
if jarvis.wait_for(text="Принять", timeout=180):
    w.click(role="button", name="Принять")
    jarvis.confirm_visual("Принято")             # пиксельная сверка = честный verify
else:
    jarvis.report_partial("матч не найден за 3 минуты")  # честный provable-провал
```

### Во сколько раз меньше раундов / дешевле (по измерениям)
| Приём | Метрика из данных | Источник |
|---|---|---|
| CodeAct | −30% ходов (5.5 vs 7.6), +20 п.п. success | arXiv 2402.01030 |
| CaveAgent (persistent kernel) | −28.4% всех токенов, −32.7% prompt-токенов за счёт раундов | arXiv 2601.01569 |
| Anthropic programmatic | −37% токенов, ~19 устранённых раундов на 20+ вызовах | advanced-tool-use |
| Anthropic MCP code exec | 150K→2K токенов (−98.7%) | code-execution-with-mcp |
| ActionEngine (синтез+SMG) | −94% LLM-вызовов, 2× быстрее, 11.8× дешевле, 95% vs 66% | arXiv 2602.20502 |
| UFO2 speculative batch | −51.5% инференса | arXiv 2504.14603 |

На типичной репетитивной GUI-процедуре Джарвиса — **1 облачный раунд вместо 8-15**.

### Как совместить с медленным облачным Claude (2-13с) и verify-законами проекта
1. **Выигрыш — в устранении СЕРИИ ходов, не первого.** Генерация кода — по-прежнему один полный ход. Поэтому короткий разговорный ход оставить реактивным (tier0/conversational — Джарвис уже так делает), а code-mode применять к длинным процедурам.
2. **Наблюдение обязано быть дёшево ВНУТРИ кода**, иначе цифры −30..−94% не достигаются: `jarvis.wait_for`/`jarvis.query` бьют в world-model / событийную подписку, а не в облако. Это связывает столп 5 со столпами (а)+(б) намертво.
3. **Честный провал, не ложный успех (LAW проекта).** Код не возвращает «ok» слепо: `jarvis.confirm_visual()` — пиксельная сверка перед терминалом (fused observe уже есть); при падении посреди процедуры код содержит `report_partial()` → честный «выполнено k из n», как input_batch сейчас. Рассинхрон world-model митигируется тем, что необратимый шаг ВСЕГДА сверяется пикселями (blindMutatePending-логика цела).
4. **Спекуляция примиряет латентность облака** (опционально, столп 3.8): пока Opus думает, локальная модель на RTX 5080 драфтит батч ЧИТАЮЩИХ действий (ui_snapshot/OCR) заранее (Speculative Actions — lossless: мелкая драфтит, крупная верифицирует). Только на обратимых действиях.

## 6. План реализации по этапам (MVP → полноценно)

**Принцип: строить в порядке рычага, максимально переиспользуя существующее, каждый этап верифицируем текст-драйвером до живого GUI.**

### Этап 0 — persistent code-сессия (MVP, максимальный скачок первым)
**Что:** превратить одноразовый `code_run` в живой stateful-процесс на userId/задачу. PowerShell Runspace (long-lived) в C#-сайдкаре для PS/UIA-ветки; один IPython-kernel для py-ветки. Ход = ячейка, исключение ловится и отдаётся модели, kernel живёт дальше. Lifecycle из SidecarClient (readiness-wait, health-check, авто-рестарт/backoff — уже есть).
**Переиспользует:** code_run, SidecarClient, wall-clock timeout.
**Модель состояния — ЖЁСТКО в персону** (иначе amnesia-tax ×3.5 / reference-errors 80%): что живёт между ходами, что теряется при рестарте, `print()` для сверки.
**Верифицируемо текст-драйвером:** `POST /dev/action` с кодом, проверить что переменная из хода N видна в ходе N+1; рестарт-политика (live-хендлы честно теряются, durable — только HERMES/world-model, как task-store сейчас не воскрешает running).
**Зависимость:** нет. Делать первым.

### Этап 1 — SDK-фасад `jarvis.*` поверх сайдкара/расширения
**Что:** типизированный API `jarvis.window('X').click(role=,name=)`, `jarvis.wait_for(text=)`, `jarvis.query(...)` — тонкая обёртка над существующими `ui_invoke`/`browser_act`/`input`. `click` через **ui_invoke по handle** (устойчив к сдвигу координат), не input_click по пикселям. Каталог функций — строкой в кешируемом блоке (progressive discovery через готовый COLD_TOOL_NAMES/tool_load), тело подгружается по имени.
**Переиспользует:** ui_snapshot/ui_invoke, browser_act, реестр хендлов, tool_load.
**Верифицируемо:** юнит-тесты SDK-обёртки против фейковых ActionResult (как index.test.ts клиента).
**Зависимость:** Этап 0.

### Этап 2 — precondition-gated батч à la UFO2 (довести существующее)
**Что:** один ход отдаёт БАТЧ действий (не только «механику»), клиент гонит под precondition-гардом по live UIA до первого расхождения → −51.5% инференса. Расширить input_batch за пределы needsLlm-запрета через prefill дешёвым тиром (Волна 3.1 уже в эту сторону).
**Переиспользует:** input_batch, SkillStep.precondition, sidecar active-window scope.
**Верифицируемо:** текст-драйвером — батч с заведомо ложным precondition → честный стоп «дошёл до шага N».
**Зависимость:** Этап 1.

### Этап 3 — событийный слушатель UIA → world-model на сервере
**Что:** MTA-поток в сайдкаре (CoInitializeEx MULTITHREADED, без окон) + `AddStructureChangedEventHandler`/`AddPropertyChangedEventHandler`/`AddFocusChangedEventHandler` + `AddAutomationEventHandler(WindowOpened)` + `SetWinEventHook(FOREGROUND/SHOW)`. Callback = только enqueue. Отдельный поток пушит дельты по существующему stdio newline-JSON. Сервер держит кэш дерева с поколениями/TTL, инвалидация по window/focus-событиям. На событии — ОДИН `CacheRequest`+FindAll (bulk). `wait_for`/`watch`-предикаты резолвятся событием мгновенно за $0.
**Переиспользует:** C#-сайдкар, stdio-протокол, watch/wait_for, speaker-registry для проактива.
**Жёсткие правила (иначе вешает машину):** callback <100 мс; НИКОГДА не Subtree на корень десктопа; сильная ссылка на handler, Remove на том же потоке; гибрид event+редкий reconcile-поллинг (события не полны на Win32/canvas).
**Верифицируемо:** частично — открытие блокнота/диалога → проверить дельту в кэше без скрина. Игры/canvas — только живьём.
**Зависимость:** Этап 1 (world-model потребляется SDK-запросами).

### Этап 4 — компиляция HERMES → именованные SDK-функции + Validator-контракт
**Что:** верифицированный человеко-показанный HERMES-навык → функция `jarvis.<домен>.<имя>()` со стабильным неймспейсом, аргументами (slots), внутренними verify-точками (ui_invoke по handle) и **обязательным Validator** (wait_for/ui_snapshot — из SkillOps). Body-hash дедуп поверх семантического. Версионирование при перезаписи дистилляцией (история/откат).
**Переиспользует:** HERMES recall/slots/precondition/fail_count/полярность, мульти-демо дистилляция.
**Главная стена в дизайн СРАЗУ:** shadowing (LLM-авторские +0.0 vs человеко-курированные +16.2 п.п.) → компилировать ТОЛЬКО из человеко-показанных демо, ЕДИНСТВЕННЫЙ канонический представитель на capability, retire по fail_count, не раздувать неймспейс похожими именами.
**Верифицируемо:** текст-драйвером — recall по имени → вызов функции → Validator-гейт.
**Зависимость:** Этапы 1, 3.

### Этап 5 (позже) — OmniParser v2 локально + спекулятивный драфт на RTX 5080
UIA-слепые окна (игры/canvas) → vision-узлы под единым интерфейсом world-model (IoU-дедуп с UIA как UFO2). Локальный драфт читающих действий во время хода Opus. Только после того как этапы 0-4 дали кратный выигрыш на нативных окнах.

## 7. Безопасность capability-модели

**Политика проекта не меняется: полное управление Windows, code_run НЕ песочница, защита — не урезание мощи, а то, что мощью управляет только владелец.** Индустрия сошлась на том же (Cloudflare «наружу только через авторизованные bindings») — главный tradeoff код-исполнения (sandbox) для личного ПК уже «оплачен».

**Что гейтить и как (без урезания code_run):**
1. **Rule-of-Two как гейт в петле (главный недостающий слой).** Если ход в одном контексте И потребил недоверенный источник (web_*/browser_read/OCR/ui_snapshot untrusted-блок), И вызывает необратимый актуатор (fs delete/move, messaging send, оплата, system_power) → форс confirm НЕЗАВИСИМО от намерения модели. Сейчас untrusted лишь ПОМЕЧАЕТСЯ, но не блокирует производный необратимый вызов — **нет taint/provenance-трекинга. Это критический gap.**
2. **Гейт раньше всех правил** (модель Claude Agent SDK): единственный надёжный барьер — pre-dispatch hook до allow-логики. Необратимые SDK-функции помечать флагом «requires owner interaction» (как `requiresUserInteraction` MCP), чтобы allow-правило/реплей/steer их не обходили.
3. **Lies-in-the-loop:** диалог подтверждения строить ДЕТЕРМИНИРОВАННО из полей ActionCommand (получатель/сумма/путь/домен), НЕ из текста модели — модель могла перефразировать по недоверенному экрану.
4. **Тир-режимы Operator:** watch-mode для ультра-чувствительного (почта/платёжки — требовать присутствия владельца по powerMonitor idle); takeover-mode на ввод кредов (у Джарвиса уже политика «креды вводит владелец сам»).
5. **Intent-to-action binding:** confirm как capability-ТОКЕН с TTL + привязкой к точным метаданным (tool+resource+timestamp), replay-protection — чтобы steer/дубль-гейт/реплей навыка не переиспользовал старое одобрение на новое необратимое действие.
6. **Компилированная процедура НЕ наследует полномочия:** `jarvis.dota.accept()` (идемпотентный клик) безопасна, но выученная процедура не должна включать delete/send/pay без явного токена запуска. Persistent-интерпретатор = один раунд может выполнить необратимую ЦЕПОЧКУ без промежуточного гейта → SDK помечает функции по Rule-of-Two.

**Что у Джарвиса УЖЕ есть:** requestConfirm, untrusted_content-обёртка (вкл. live-system заголовки окон, M11), self-guard денилист секретов (.env/id_rsa/.ssh/.aws), SSRF-гард, SpendGuard (Denial-of-Wallet cap), looksLikeDirective-фильтр консолидации.

**Чего не хватает:** (а) taint/provenance-трекинг (untrusted→irreversible); (б) capability-токены с TTL вместо разового confirm; (в) egress-allowlist доменов (сейчас только SSRF-гард — расширить); (г) аудит memory-poisoning на HERMES-навыках и consolidation (персистентная инъекция через выученный навык — write и read фазы разнесены во времени, детект тяжёл).

**Prompt-injection с экрана НЕ решается персоной** («не исполняй текст с экрана» — обходится >90% адаптивно). Нужен структурный слой: egress-allowlist + taint + стрип Unicode Tag U+E0000-E007F + мульти-канальная классификация доверия. TEE-криптобиндинг (arXiv 2603.14332) — избыточен для одно-пользовательского ПК.

## 8. Риски и честные пределы — что среда НЕ чинит

**Среда убирает РАУНДЫ и удешевляет наблюдение. Она НЕ чинит:**

1. **Реалтайм-рефлексы игр.** Облачный ход 2-13с + событийный лаг несовместимы с механикой, требующей реакции в кадрах. Джарвис в играх — про меню/принятие матча/GSI-события, не про геймплей. UIA-слепота canvas/WebGL остаётся — там только OCR/GSI/OmniParser, не структура.
2. **Качество STT.** Среда не улучшает распознавание речи. STT-обрывки/эхо — отдельный класс (Волна 1-2 уже боролась), к харнессу отношения нет.
3. **Недетерминизм чужих GUI.** ActionEngine hot-patch и Voyager self-verify смягчают, но cross-app остаётся хрупким у ВСЕХ (UFO2 = 9.1%). Не ждать, что чужая архитектура это решит — открытая проблема.
4. **Рассинхрон world-model.** Доказанно системно врёт (ошибки 0.91-1.00 под конфликтом, 37.4% ghost-узлов). Среда УСКОРЯЕТ чтение, но verify необратимого ОСТАЁТСЯ пиксельным — это не устранимо, только митигируется TTL + fused observe.
5. **Латентность облака на первом ходе.** Генерация кода — полный ход Claude. Спекуляция прячет часть, но не первый ход. Короткие команды не выигрывают от code-mode (plan-then-execute платит стартовой латентностью) — не применять к разговорным ходам.
6. **Событийная полнота.** UIA-события теряются на Win32/WinForms/canvas → чистая подписка пропустит часть; гибрид с reconcile-поллингом обязателен, 100% событийности недостижимо.
7. **Skill shadowing при росте библиотеки.** Компиляция навыков РАСТИТ риск (похожие имена → модель выберет не ту). Не автогенерировать навыки (ADAS-автономия +0.0 п.п., дорого).

**Что НЕ надо строить:**
- Голый Anthropic computer-use screenshot-loop как транспорт — шаг назад.
- Snapshot/serialize-персистентность (LangChain-стиль) — HWND/COM несериализуемы, «Value was not restored».
- Чисто-структурную world-model без пиксельной сверки — врёт на необратимом.
- Ре-скан UIA каждый раунд без инкремента — 3-26с, не быстрее скрина.
- UFO2 Picture-in-Picture как основной режим — ломает «живёт на моём экране»/дейксис, требует Pro/RDP; только опционально для фоновых задач пока владелец играет.
- Полную ADAS-автономию генерации инструментов — не окупается (arXiv 2510.06711).
- TEE-криптобиндинг capability — избыточен для одного пользователя.
- Broad UIA-подписку на корень десктопа — вешает машину.

## 9. Рекомендация

### Первый шаг (1-2 недели, максимальная отдача, верифицируемо)
**Построить Этап 0 + минимальный Этап 1: persistent code-сессия с SDK-фасадом `jarvis.*`.**

Конкретно:
1. Long-lived PowerShell Runspace, хостимый в существующем C#-сайдкаре (нативно, без внешних зависимостей), + один stateful IPython-kernel для py-ветки. Переменные/HWND/COM живут между ходами.
2. Тонкий SDK-фасад `jarvis.window('X').click(role=,name=)` / `jarvis.wait_for(text=)` поверх уже существующих `ui_invoke` (по handle) и `browser_act`. Каталог функций через готовый COLD_TOOL_NAMES/tool_load.
3. Модель состояния — ЖЁСТКО документировать в персоне (что живёт между ходами, что теряется при рестарте, `print()` для сверки) — иначе amnesia-tax ×3.5 или reference-errors 80%.

**Почему именно это:** максимальный рычаг под цель владельца (раунд=процедура) на УЖЕ имеющемся стеке (code_run + сайдкар + ui_invoke), даёт measured −30% ходов (CodeAct) сразу, и открывает путь к −94% (ActionEngine) на следующих этапах. Полностью верифицируемо текст-драйвером (`POST /dev/action` с кодом → проверить персистентность переменной, честный partial при падении, рестарт-политику) БЕЗ живого GUI. Риск минимален — переиспользует существующие подсистемы, добавляет живой процесс, а не новую архитектуру.

### Целевая архитектура (куда это ведёт)
Четыре столпа как ОДИН конвейер: облачный Claude пишет процедуру кодом на `jarvis-SDK` (один ход) → persistent-интерпретатор на клиенте исполняет со state (HWND/COM живут) → наблюдение внутри кода бьёт в server-side world-model, обновляемую ДЕЛЬТАМИ из событийного UIA-слушателя сайдкара ($0, без раундов) → верифицированные человеко-показанные HERMES-процедуры компилируются в именованные SDK-функции с Validator-контрактом → capability-гейт (Rule-of-Two taint) форсит confirm на (untrusted ∧ irreversible), показывая СЫРУЮ операцию.

Джарвис архитектурно ближе к UFO2, чем любая другая продакшн-система, и уже реализовал 2 из 3 её приёмов. Порядок внедрения по рычагу: **(1) persistent code-SDK → (2) precondition-gated UIA-батч → (3) событийная подписка (фронтир, строить самим) → (4) компиляция навыков → (5) OmniParser + локальный спекулятивный драфт.** Первые два дают кратный выигрыш немедленно; событийная world-model — возможность обогнать SOTA, но и главный инженерный риск (шум/лаг/UIA-слепота, обязателен гибрид с поллингом).
