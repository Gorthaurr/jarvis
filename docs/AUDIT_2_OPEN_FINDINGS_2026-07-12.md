## [9] messaging: дубль отправки при «доставлено-но-таймаут» (design-tradeoff)

**Файл:** `apps/server/src/brain/tools/handlers/messaging.ts` (`telegramSend`, ключ ставится только после `result.ok`).

**Суть.** Идемпотентный ключ (`sentKeys.set`) фиксируется ТОЛЬКО в ветке `result.ok`. Если сообщение
РЕАЛЬНО доставилось в web.telegram, но ack потерялся/пришёл позже 90с → `sendAction` вернул `!ok`
(timeout) → ключ не поставлен → следующий заход (агент-ретрай ИЛИ фолбэк на расширение в том же вызове)
отправляет ВТОРОЙ раз → дубль адресату. То же в `outbound.ts` (`markSent` только после `sent.ok`).

**Почему не исправил автономно.** «Резервировать ключ ДО отправки» рискует заблокировать ГЕНУИННО
провалившуюся отправку (не доставилось → но ключ стоит → сообщение никогда не уйдёт). Это реальный
UX-компромисс на денежном/сообщенческом пути, и он НЕ верифицируется без живого Telegram/клиента.
Частично уже смягчено фиксом [8] (дедуп теперь ОКНО `SEND_DEDUP_MS`, а не «навсегда»).

**Рекомендованный фикс (на твоё решение).** Паттерн «reserve-then-release»: ставить ключ в окне ДО
отправки; на ОПРЕДЕЛЁННО-непоставку (`channel_down` / resolve-issue — команда точно не дошла) — снимать
ключ (`delete`), разрешая ретрай; на неоднозначный таймаут — ключ ОСТАВЛЯТЬ (в окне не дублируем).
Плюс убрать авто-фолбэк на расширение при таймауте CDP (иначе тот же вызов шлёт дважды). Идеально —
delivery-confirmation от клиента (id пузыря), но это отдельная работа.

## [10] office: таймаут одного COM-вызова убивает ВСЕ headless-инстансы (concurrency-kill)

**Файл:** `apps/client/main/actuators/office.ts` (`killOfficeTree`, ~стр.224).

**Суть.** Office НАМЕРЕННО вне аренды ввода (`input-kinds.ts`: «отдельные COM-инстансы за курсор не дерутся»)
→ два `office_excel` над РАЗНЫМИ файлами идут параллельно через Semaphore. Если один зависает на 30с →
`killOfficeTree` делает `Get-Process EXCEL | Where MainWindowHandle -eq 0 | Stop-Process -Force` — снимает
И невидимый Excel второго (исправного) вызова посреди `Save()` → потеря данных / «Office COM ошибка» на
операции, которая была здорова. `withPathLock` защищает только один-и-тот-же файл, не кросс-файловую гонку.

**Почему не исправил автономно.** Заскопить kill на ОДИН зависший инстанс сложно: Office-сервер не
дочерний процесс powershell (COM-активация), PID именно того EXCEL.EXE не отслеживается. И это COM —
без живого Office не проверить. Плюс сериализация Office меняет намеренную concurrency-модель (твой выбор).

**Рекомендованный фикс (на твоё решение).** Либо (а) сериализовать ВСЕ Office-COM вызовы одним мьютексом
(тогда kill-by-image всегда бьёт только текущий-единственный инстанс) — ценой параллелизма; либо (б)
запускать каждый Office-вызов в отдельном PS-процессе, ЗАПОМИНАТЬ его PID и убивать ТОЛЬКО его дерево
(а не все EXCEL по имени образа). (б) точнее, но больше работы.

## [12] office: append_row затирает B1, если A1 пуста (edge-case, потеря данных)

**Файл:** `apps/client/main/actuators/office.ts` (`append_row`, проба пустоты ~стр.64).

**Суть.** Проба «лист пуст» смотрит на АБСОЛЮТНУЮ `A1` (`$ws.Cells.Item(1,1)`). Лист, где заполнена только
`B1` (а `A1` пуста): `UsedRange.Rows.Count==1`, `A1` пуста → `$empty=$true` → `$row=1` → append пишет в
`A1,B1,C1…`, ПЕРЕЗАПИСЫВАЯ существующую `B1` вместо добавления новой строки. `ok=true` при потере данных.

**Почему не исправил автономно.** PowerShell/COM без тест-харнесса (нельзя прогнать без живого Excel), а
data-loss-фикс вслепую рискованен. Фикс несложный, но требует живой проверки на реальном файле.

**Рекомендованный фикс (на твоё решение).** Определять номер строки для дозаписи по `UsedRange`:
`$row = $ws.UsedRange.Row + $ws.UsedRange.Rows.Count` (первая строка ПОСЛЕ занятого диапазона), а не по
пустоте абсолютной `A1`. Для полностью пустого листа (`UsedRange` = A1, ячейка пуста) — оставить `$row=1`.
