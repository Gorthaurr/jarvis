## 3. Память: почему facts:0 — диагноз

Не «модель ленится». **Три структурных разрыва**, каждый самодостаточно даёт facts:0:

### Разрыв 1 — profile.facts: стор без писателя
`brain/profile.ts:119 addFact` — ноль продакшн-вызовов (grep: только `profile.test.ts`). В `data/profile.json` ключа `facts` физически нет. При этом именно `p.facts` читают: контекстное приветствие (`server.ts:799` → `greeting.ts:59` — осознанно ТОЛЬКО курируемые факты) и промпт (`router-ws.ts:243`). **Единственный потребитель курируемых фактов читает стор, куда не пишет никто.**

### Разрыв 2 — episodic: писатель есть, но модель его не зовёт
`memory_write` (горячий, схема видна каждый ход) — **0 вызовов за 15 дней** логов (07-04, 07-10: 0/0 при 57 tool-вызовах/день). Последняя запись в БД — 2026-06-25. Причина в персоне: `persona.md:485-489` «Better to under-record than pollute» — правило чисто запретительное, позитивного триггера «КОГДА писать» нет, механики-бэкстопа нет. Контраст с навыками показателен: у процедур есть авто-петля `selfLearnSkill` (`agent/index.ts:1636-1660`) → 29 learned-навыков на диске. **Джарвис выучил КАК делать, но не выучивает КТО его хозяин.**

### Разрыв 3 — даже наполненные facts затеняются спредом
`agent/index.ts:733-735`: `buildSystemPrompt({...deps.userContext, facts, ...})` — ключ `facts` (результат episodic.search, сейчас всегда `[]`) стоит ПОСЛЕ спреда и **безусловно перезаписывает** профильные факты из `deps.userContext`. Мина: любой фикс разрывов 1–2 не даст эффекта, пока эта строка не починена.

### Отягчающие
- **Стор загрязнён:** 142 event (STT-мусор июня: «Усни.», «Гарвис не наем») против 13 fact, среди которых 5 дублей и мета-костыль «игнорировать шум» (модель чинила мусор записью в тот же стор). `stale` есть в схеме — никто не ставит `stale=true`, дедупа на записи нет (у skills есть — `findDuplicateSemantic`).
- **Порог не калиброван:** `JARVIS_MEMORY_MIN_SCORE=0.78` при сжатой шкале e5 (нерелевант 0.72–0.80, релевант 0.82–0.92) — посередине; `episodic.ts:48-52` честно признаёт отсутствие калибровки.
- **Голосу — 350мс:** `ioTimeoutMs = sink ? 350 : 700` (`agent/index.ts:694`) накрывает episodic.search + skills.recall + catalog; провал — log.debug, в форензике невидим.
- **Не та задача:** единственный вход фактов — query-зависимый top-5 по ТЕКУЩЕЙ фразе. Поиск по «включи музыку» никогда не вытащит «работает по ночам». Стабильный профиль и эпизодический recall — разные потребности, склеенные в один механизм.
- **Нет консолидации:** разговоры старше 40 реплик/12ч теряются навсегда — «сон-цикла» working→episodic нет, хотя все кирпичи (дешёвый тир, e5, episodic.write, scheduler-паттерн) в наличии.

## 4. Находки из вечерних логов (2026-07-10, вечер Доты)

1. **Wake-гейт потерял 2 реальные команды молча.** Из 12 «реплика без обращения — игнор»: 9–10 — игровой трёп (гейт прав), но «**Дарья**, запусти поиск в доте» (16:13:40, lev=4 от «Джарвис» — оба слоя матчера бессильны) и «Гуляю, почине героя» (16:22:12, почти наверняка «Джарвис, пикни героя» — повтора не было, **пик так и не сделан**). `DEEPGRAM_KEYTERM` выключен (`deepgram.ts:112-117`), near-miss не логируется — дроп неотличим от трёпа.
2. **STT-ход длиной 39 минут.** Energy-VAD без потолка (`vad/index.ts:39-43`): игра из колонок держит RMS выше порога → speech_end не наступает часами. Мега-ход 15:32→16:11 = 116 851 кадр; за день 452 пустых транскрипта против 3 с текстом; достоверных финалов за вечер — 2. Команда внутри такого хода не финализируется в принципе.
3. **Онбординг ×7 за день, без кулдауна.** Гейт только `!resumed` (`server.ts:785`), registry в ОЗУ → каждый рестарт сервера/клиента = приветствие (одно — посреди игры, 16:19:43). 2–3 из вечерних — dev-сессии текст-драйвера (тоже здороваются, слушателя нет). Каждое = haiku-вызов + TTS. При этом «контекстное» приветствие читает мёртвый profile.facts — контекста в нём никогда не было, кроме имени и времени суток.
4. **Провал resume теряет управление задачами.** Голосовой «отмени» ищет задачи по `sessionId` (`task-control.ts:66`), задачи привязаны к СТАРОМУ → «Отмени поиск в доте» **дважды породил новую фоновую задачу** (Sonnet-петли, одна с эскалацией на Opus, recall нерелевантного навыка «пикнуть героя» sim 0.85 на команду отмены); убивать пришлось кнопками UI. Также теряются: pendingClarify, toolActivation §15, awake-окно голоса.
5. **Задачи-зомби на мёртвом сокете — ~$0.68 впустую.** После обрыва канала (16:14:54) петли ~2 мин слали команды в закрытый сокет: ×3 ActionCommand timeout 15с, 2× эскалация на Opus «лечила» отсутствие сокета, три задачи ok:false (146с/142с/78с).
6. **Комплимент за $0.19.** «Джарвес, да ты молодец» → conversational-маршрут, но полный tool-loop: 8 раундов, screen_capture ×3, 35с, регистрация §20-задачей «Да ты молодец» (участвовала в scope-решениях следующих реплик).
7. **client.system — слепая зона наблюдаемости.** Хендлер (`router-ws.ts:459-478`) не пишет ни строки лога; клиент при пустом снимке молча не шлёт кадр. По логу невозможно доказать, что live-контекст вообще доезжал.

## 6. Что НЕ делать

1. **Не класть live-контекст в кешируемый префикс.** Всё переменное (окна, вкладки, занятость, время, retrieval-факты) — только `systemDynamic`/хвост messages. Каждая правка персоны/tools = перезапись 33.5K (~$0.10–0.20 холодного хода) — батчить.
2. **Не пересобирать system-блок per-round.** Rolling-брейкпоинты в messages инвалидируются сменой system — рефреш только append'ом в хвост (Б3). Это класс Д5 из HARNESS_PLAN.
3. **Не превращать память в дамп.** Принцип «осознанной памяти» верен — мусорный стор июня (142 обрывка STT) это доказал. Чинить не снятием запрета, а: позитивные триггеры + бэкстоп-нудж + консолидация с анти-мусорным промптом + дедуп + stale. Кап 5 фактов/день, кап 20 в профиле. Никакой авто-записи сырых реплик.
4. **Не включать экстрактор до порога и гигиены.** memoryMinScore с дефолтом 0 + top-5 без отсечения = «вспоминает то, чего не было» в первый же день наполнения. А9 (stale-миграция, калибровка) и Б2-порог — В ОДНОМ изменении с Б1.
5. **Не строить новые сенсоры.** enumWindows, WASAPI, powerMonitor, реестр, Steam-манифесты — уже собирают всё нужное для уровня А. План довозит собранное до промпта; новые пробы (SMTC) — on-demand инструментом, не per-tick.
6. **Не дублировать Волну 2.** `ui_snapshot`/`windows_list` — on-demand инструменты по HARNESS_PLAN; здесь — проактивная сторона (что модель видит БЕЗ вызова инструмента). Границу держать: проактив = дешёвая сводка ≤650 ток, полнота — инструментом.
7. **Не ослаблять wake-гейт огульно.** 9–10 из 12 вечерних игноров — правильные (игровой трёп). Second-chance — только узкий (императив + лексика активной задачи + уточняющий вопрос), не снижение порога fuzzy.
8. **Не «лечить» транспорт эскалацией тира.** Мёртвый сокет ≠ слабая модель; Opus на слепых раундах — чистый сжиг денег (§4.5).
