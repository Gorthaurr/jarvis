## 1. КАК ЗАПУСКАТЬ
- **Сервер** (мозг, Fastify+WS, порт **8787**): из `apps/server` → `npx tsx src/index.ts` (НЕ `tsx watch` — падает EADDRINUSE).
  Перезапуск: убить PID на 8787 (`taskkill /F /PID <pid>`) → запустить заново. Логи → `server.out.log`/`server.err.log` (редирект).
  Грузит `.env` из корня репо. Проверка живости: `curl http://127.0.0.1:8787/healthz` → `{"ok":true,...}`.
- **Клиент** (Electron, UI/микрофон): `apps/client` → `node scripts/build.mjs` (esbuild) → `pnpm start` (electron). После правок `.ts` клиента — пересборка. Electron запускать БЕЗ редиректа stdio.

- **БД**: нативный PostgreSQL + pgvector (`DATABASE_URL`); фолбэк PGlite. Docker НЕ используется.
- **Тесты**: `apps/server` `npx vitest run` (~613); `apps/client` `npx vitest run`. Typecheck: `npx tsc --noEmit` или `pnpm -r typecheck`.

## 2. ⭐ КАК Я ТЕСТИРУЮ БЕЗ ПОЛЬЗОВАТЕЛЯ (важнейшее — раньше не знал)
**Я могу гнать команды прямо в Джарвиса ТЕКСТОМ и смотреть результат сам — голос не нужен.**

### 2a. Текст-драйвер (главный инструмент)
В корне репо есть `_jarvis_cmd.mjs` — WS-клиент, который шлёт `dev.text` в агента (как печать в чат) и печатает ответы.
Протокол: WS на `ws://127.0.0.1:8787/ws` → `client.hello{token,clientVersion,protocolVersion:1}` → ждать `server.hello` →
слать `dev.text{text}`. **ОБЯЗАТЕЛЬНО** отвечать `pong` на `ping` (иначе heartbeat закроет сессию). Авто-подтверждать
`user.confirm.request`. На `action.command` (для клиент-актуаторов) отвечать фейковым `action.result{ok:true}` (telegram/web/MCP идут МИМО — server-side/расширение, поэтому реальны).
```
node _jarvis_cmd.mjs 'отправь голосовое в избранное с текстом тест' 'да, отправляй'
```
Node 22 имеет встроенный `WebSocket` — зависимостей не надо.

### 2b. Что РЕАЛЬНО, а что фейк в текст-драйвере
- **Реально** (server-side / через расширение): web_search, memory_*, MCP-инструменты, set_reminder, telegram_* (через ExtensionBridge), сам LLM/persona/маршрутизация.
- **Фейк** (нужен живой Electron-клиент): app_launch/fs/input/system/screen — это `action.command` к клиенту; мой драйвер отвечает `ok` вслепую. Их живьём проверяет только запущенный клиент.
- Аудио (TTS) идёт `speak.chunk` — игнорирую; смотрю `chat`-сообщения (текст реплики).

### 2c. DEV-эндпоинты (прямая проверка расширения, минуя агента)
- `POST /ext/telegram {to,text}` — отправить ТЕКСТ в Telegram напрямую (чистый тест расширения).
- `POST /ext/telegram_file {to,audioB64}` — отправить АУДИО-файлом (base64 mp3).
- `POST /ext/reload` — перечитать расширение.
- Кириллица в теле curl на Windows ломает Content-Length → писать тело в файл и `--data-binary @file.json`.

### 2d. Логи — источник истины
`server.out.log`: маршрутизация (`tier`), `tool { name, isError }`, `prompt-кеш (§15)`, `latency:`, дамп ошибок расширения
(`telegram-file: step: ... | DOM={...}` — там видно состояние webK). Читать через Read/grep (`grep -a`), не PowerShell (кириллица в пути `Автокомп` ломает).

## 4. АГЕНТ-ПЕТЛЯ И ЛАТЕНТНОСТЬ
- **Тиры** (`brain/router`): haiku=короткая болтавня/счёт/знание → СИНХРОННО; sonnet/fable=многошаговая/рассуждающая → ФОН
  (мгновенный дворецкий-ack «Сию минуту/Извольте», результат по готовности через `speakResult`). **ВСЕ тиры = модель `claude-opus-4-8`** (в `.env`) — эскалация тира пропускается, т.к. модель одна.
- **Латентность (замерял текст-драйвером):** болтовня/счёт **~2с**, содержательный ответ **~6с**, задача с инструментами
  **~8–15с** (маскируется мгновенным ack). Цель 800мс на тяжёлом ходе не берётся — это норм для Opus.
- **Prompt-кеш §15 РАБОТАЕТ:** префикс [персона][каталог-инструментов][навык] кешируется; повторные ходы — `cacheRead`
  (дешевле/быстрее). Любая мутация `tools[]` рушит кеш → ленивая загрузка холодных инструментов (`tool_load`).
- **max_tokens** — это ВЫВОД за ход (не контекст). Обрыв докручивается (continuation). НЕ слать temperature/top_p/top_k на Opus (HTTP 400). thinking слать МОЖНО — но Opus только `{type:"adaptive"}` (enabled+budget → 400); adaptive = рабочее рассуждение, не убирать.

## 5. ПАМЯТЬ / ЗАДАЧИ / CONTINUITY (главная боль — починена)
- **Working memory** (`memory/working.ts` + `working-store.ts`): окно диалога (40 реплик), персист `data/memory/<user>.json`
  — переживает рестарт сервера/клиента и обрыв WS. Resume-окно 120с (`registry.scheduleRemove`).
- **Реестр задач §20** (`brain/tasks/manager.ts` + `task-store.ts`): персист `data/tasks.json` (атомарно, flush на close).
  - `recentTerminal(userId)` — последние СОДЕРЖАТЕЛЬНЫЕ (`stepsDone>0`) терминальные задачи → ответ на «сделал?».
  - `activeForUser(userId, excludeId)` — задачи В РАБОТЕ → ответ «ещё считаю», а не «ничего не делаю».
  - На restore НЕ-терминальная задача → честный `failed` («прервано перезапуском»), не воскрешается.
- **Инжект** обоих блоков → `formatActiveTasks`/`formatRecentTasks` → `UserContextSlot.recentTasks` → `renderDynamic`
  (НЕкешируемый хвост, кеш §15 не страдает).
- **freshContext (важно!):** для «обособленной» новой задачи во время активной агент берёт КОРОТКОЕ окно (`recentTurns(10)`),
  а НЕ ноль — иначе вопрос-продолжение («ты отправил?») терял весь диалог и отвечал «не вижу о чём речь». Память = ВСЕГДА несём окно диалога.
- Episodic (pgvector) — память ОСОЗНАННАЯ: пишем `memory_write` только значимое, НЕ каждую реплику (иначе «странные графики на старте»).

## 7. РУКИ В БРАУЗЕРЕ / РАСШИРЕНИЕ / webK (где я долго буксовал)
- **ExtensionBridge** (`gateway/extension-bridge.ts`): сервер ↔ расширение по WS `/ext`. `request(intent, timeout)` ждёт ответ
  по id. ⚠️ `rejectAllPending` на каждом reconnect/disconnect — поэтому длинная операция, пережившая обрыв WS, ОТКЛОНЯЕТСЯ.
- **MV3 service worker** (`apps/extension/background.js`): засыпает при простое >30с между событиями (alarm clamp ≥30с) →
  WS рвётся → «расширение не ответило». **Фикс: `startKeepAlive()`** — пинг `chrome.runtime.getPlatformInfo` каждые 15с во
  время операции (telegramSend/telegramSendVoice). БЕЗ него длинная файловая операция падала.
- **webK (web.telegram.org/k/) рендерится ТОЛЬКО когда вкладка ВИДИМА** (`visibilityState:visible`). Фоновое/скрытое окно
  → Chrome ставит rAF на паузу → webK висит на `has-auth-pages` с пустым телом (выглядит «не залогинен», хотя сессия есть).
  **Фикс: `openTgTab()`** — переиспользуем УЖЕ открытую вкладку пользователя (DOM построен) ИЛИ активируем её
  (`tabs.update active + windows.update focused`) на время отправки; нет вкладки → создаём ВИДИМОЕ окно (focused:true).
- **Открытие чата:** клик по строке `.chatlist-chat` + НАТИВНЫЙ `.click()` (webK навигирует по href), ждём композер.
  Интерфейс может быть EN → «Saved Messages», не «Избранное» (матчить оба).
- **ТЕКСТ в Telegram — РАБОТАЕТ** (проверено `/ext/telegram`). **АУДИО-ФАЙЛ — СТЕНА:** webK блокирует программную загрузку
  файлов (синтетический `input.change` и drag-drop игнорируются — защита). Настоящее голосовое — только GramJS (нужен api_id,
  которого Telegram-цензура у Антона не даёт: my.telegram.org режет создание app с РФ-IP, страна номера≠IP).
- `browser_open/read/act` — в реальных вкладках через `chrome.scripting` (мышь не трогаем). `browser_inspect` — глаза в DOM.

## 8. ГОЛОС / ДИКТОР-ГЕЙТ / TTS
- **Гейт диктора (`voice/pipeline.ts` + `speaker/`): СЕЙЧАС ВЫКЛЮЧЕН** (`JARVIS_SPEAKER_GATE=0` в `.env`) — биометрия
  ложно отклоняла голос ВЛАДЕЛЬЦА (скоры 0.03–0.7, порог 0.35), Джарвис «не реагировал». Включается, когда `!= "0"` И есть
  enrolled-голос. Owner-only вернуть только после переработки биометрии (банк векторов + AS-Norm), иначе снова отклоняет своего.
- **Wake «Джарвис»** (`voice/wake.ts`): MockWakeWord (текстовый, не акустика). Fuzzy-матч ослышек STT (Гарвис/Джервис/Jarious/jarvis…).
- **TTS = Yandex, голос filipp** (`TTS_PROVIDER=yandex`). filipp эмоцию good/evil НЕ умеет (только strict). Аудио-теги `[warmly]` Yandex срезает.
- **Гигиена вывода (`brain/verbalize/`):** `verbalize()` чистит текст ПЕРЕД голосом И чатом — markdown/url/эмодзи + `stripToolCallSyntax`
  (утёкший `<invoke>`/`<parameter>`) + `stripAudioTags` (`[warmly]`). Числа→слова, латиница→фонетика.
- **STT Deepgram** (nova-3): открытие/закрытие WS каждый ход (churn) — открытая боль; persistent WS не сделан (риск).
- **§9 «Уважительная проактивность» (не мешать):** `pipeline.speakQueued(text, urgent?)` + `maybeDrainSpeech` — НЕсрочную
  проактивную речь (итоги фоновых задач) держим, пока `isUserBusy()` (из `client.context`: звонок/полный экран/блокировка),
  отдаём по `drainPending()` (router зовёт на смену контекста). Напоминания = `urgent` (будильник, проходят всегда). Клиент:
  `Sensors` (sensors/index.ts) шлёт `client.context`, реальный `locked` через powerMonitor; fullscreen/micBusy — пока дефолт false (TODO Win32).
- **Навыки HERMES хранятся в БД (skills-таблица), НЕ в .md-файлах.** `data/skills/*.md` — только артефакт для чтения; recall
  читает БД/memSkills. Чтобы научить навыку — `skill_save` (или провайдер `.save()`), не просто положить .md. Recall лексический
  (≥2 токен-хита, ≥0.34) — для коротких команд часто не срабатывает (норма). Сидинг см. сессию 2026-06-19 ч.3.
