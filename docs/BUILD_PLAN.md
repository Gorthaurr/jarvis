# BUILD_PLAN — Вертикальные срезы M0..M8 (§17)

> Главное правило: строить **вертикальными срезами**, каждый — рабочий и тестируемый.
> Не пытаться поднять всё сразу. (§17 спецификации)

---

## M0 — Скелет ✅ каркас готов

**Что входит:**
- Electron-клиент + постоянный WebSocket к серверу
- Server gateway: auth, per-user сессия, handshake/heartbeat/reconnect/in-flight (§5)
- Tier-0 детерминированные локальные команды (`app.launch`, `app.focus`) без сети, стоимость $0
- Текстовый ввод в renderer (без голоса)
- Persona-промпт + детерминированный вербализатор (§21)
- Billing limits / spend cap (§14)
- Working memory, scheduler.computeTriggerTs (§9)
- Миграции PostgreSQL (все таблицы §13) + docker-compose
- Полный типизированный протокол `packages/protocol` (§5/§6)
- Утилиты `packages/shared`, JSON-схемы `packages/tools`
- C#-скелет `sidecar-win` (интерфейс UIA + SendInput)
- Android-скелет `mobile` (геофенс + FCM)

**Тест-критерий:**
> Текстовая команда «открой блокнот» → notepad.exe запускается на Windows.
> Клиент получает `action.command{kind:"app.launch"}`, исполняет, отвечает `action.result{ok:true}`.

## M1 — Голосовой цикл 🟡 скелет

**Что входит:**
- Wake word локально (openWakeWord / Porcupine): активация стрима по «Джарвис»
- VAD (Silero): начало/конец речи, barge-in
- Аудио-стрим: захват в renderer (WebRTC AEC), гейтинг по wake word, LiveKit к серверу
- STT: Deepgram streaming (§1); bake-off на русском, проверка качества
- Turn detection: штатный semantic turn detector LiveKit (мультиязычный); тест на русском
- TTS: ElevenLabs streaming → воспроизведение в renderer; цель — первый звук <800 мс (§10)
- Barge-in: говорит пользователь → TTS рубится, перезапуск
- Follow-up окно ~6 с без повторного wake word (§10)
- Расширенный вербализатор: даты, времена, валюты, единицы по-русски (§21)
- Валидация wake word «Джарвис» по-русски (вероятно, нужна кастомная модель, §18)

**Тест-критерий:**
> «Джарвис, который час» → голосовой ответ, первый звук < 800 мс от конца фразы.
> Barge-in работает (перебивание рубит TTS).
> Follow-up «а погода?» без повторного wake word.

## M2 — Роутинг + память 🟡 скелет

**Что входит:**
- TS-роутер: classifyTier — Haiku/Sonnet/Fable-выбор по намерению; каскад эскалации (§7)
- PostgreSQL + pgvector: рабочая память (сессия) + эпизодическая (факты, предпочтения, события)
- Семантический поиск по эпизодической памяти (text-embedding-3-small, 1536d, HNSW)
- Persona: статичный кешируемый префикс + динамика юзера в первом сообщении (§11)
- Cache prefix: общий промпт для всех юзеров — кешируется глобально, скидка до 90% (§15)
- Стек сущностей для анафоры между репликами («её» → предыдущий контакт, §10)
- Server-side веб-знания: `web.search` (Brave Search / SearXNG) + `web.fetch` (readability); Q&A не трогает браузер юзера (§12)

**Тест-критерий:**
> Факт из прошлой сессии достаётся из эпизодической памяти.
> «Открой браузер» → tier-0 (без LLM); «что там нового у Anthropic?» → Haiku-классификация → Sonnet-ответ с web.search.
> UC-9: веб-вопрос без открытия GUI-браузера (§16).

## M3 — Актуаторы + грундинг 🟡 скелет

**Что входит:**
- win-сайдкар (C#/.NET) — полная реализация (самый трудоёмкий кусок, §18):
  - UIAutomation-дерево: резолв контрола по роли/имени
  - UIA-паттерны: `ui.invoke` (InvokePattern, ValuePattern, SelectionItemPattern, TogglePattern, ScrollPattern) — основной путь действия без захвата курсора (§6)
  - SendInput: синтетическая мышь/клавиатура — fallback после паттернов
  - IPC: stdio/named pipe к Electron main
- expect/auto-wait/retry в skill-runner (§6): постусловие шага → поллинг a11y до наступления
- User takeover-пауза: физический ввод во время задачи → `paused` (§20)
- `context.read(selection|active_window|screen)`: дейктические запросы (§19)
- hak-browser (puppeteer-core): веб-автоматизация, accessibility-дерево
- `code.run` с ограничениями §6: Job Object, сетевой запрет, CWD=temp, lint-гард на сервере; powershell — всегда confirm
- App allowlist (§14)
- Каркас задач: taskId + cancel-флаг перед каждым шагом (полная модель — M8)
- DPI/мультимонитор: Per-Monitor V2, виртуальные координаты (§18)

**Тест-критерий:**
> UC-4: «сделай таблицу расходов» → Fable-сессия → code.run с openpyxl → Excel открывается с данными.
> UC-3: «включи ТикТок» → hak-browser → автопереход видео без LLM в цикле.
> UC-8: «что это значит?» с выделенным текстом → context.read(selection) → ответ без скриншота всего экрана (§19).

## M4 — Скиллы 🟡 скелет

**Что входит:**
- Цикл explore→write: Fable-сессия исследует новый GUI/инструмент → на успехе записывает SKILL.md (§8)
- `packages/skills` / `memory/skills.ts`: CRUD SKILL.md per-user; `content_md` — канонический источник; `steps` — derived-парс
- **Клиентский skill-runner** (tier-0.5): детерминированное исполнение шагов без LLM в цикле; эскалация на Sonnet только при fail или needsLlm-шаге (§8)
- Verify-шаг: постусловие всей задачи (файл существует, сообщение в «Отправленных»)
- Обучение демонстрацией («смотри, покажу»): demo.record → UIA-события + голосовой комментарий → Fable конвертирует → ревью guard-шагов перед первым применением (§8)
- Ночная консолидация: Haiku по action_log → дедуп скиллов, починка повторяющихся ошибок, чистка stale-памяти (§8)
- Guard-шаги protected: консолидация не трогает `message.send`, confirm, `code.run`; новая версия скилла активируется после первого успешного прогона; авто-rollback по fail_count

**Тест-критерий:**
> Вторая попытка той же задачи идёт через skill-runner (LLM не вызывается, проверить в billing).
> Демонстрация → рабочий SKILL.md без ручной правки.
> Консолидация инкрементирует версию при успехе и делает rollback при росте fail_count.

## M5 — Проактивность 🟡 скелет

**Что входит:**
- Триггеры: время, контекст (открыл приложение X), внешние (событие календаря)
- Salience-фильтр (Haiku): «стоит ли сейчас» — busy-сигнал из `client.context` (созвон, fullscreen, lock, §9)
- Умное напоминание (§9): `intents` с дедлайном + place_id; `computed_trigger_ts = deadline − ETA − prep_minutes − buffer`; пересчёт на каждое гео-событие
- ETA: Yandex/2GIS API (авто/транспорт) или OSRM (пешком)
- Геофенс: Android Geofencing API (место → событие, а не поллинг GPS); гео-события по HTTPS на сервер
- Мобильный компаньон (Android-first): FCM data-сообщения с `expiresAt`; просроченный nudge не произносится
- Presence-роутинг: `devices.last_seen_at` определяет куда доставить — голос или пуш
- Таблица `devices`, calendar read-only (Google Calendar API / CalDAV, за `ICalendarProvider`)

**Тест-критерий:**
> UC-1: «в 9 должен быть в зале» → напоминание приходит за (ETA+сборы+буфер) до 09:00, сдвигается при смене локации.
> Salience режет nudge, если активный созвон (`micBusyByOtherApp=true`).
> Юзер офлайн → пуш на телефон; просрочен — в лог.

## M6 — Переписка ⬜ не начато

**Что входит:**
- Userbots: GramJS (Telegram) + vk-io (VK) — от аккаунта пользователя
- Сессии userbot'ов на клиенте (Electron safeStorage / DPAPI); на сервер уходят только метаданные (§12)
- Контакты: резолв «Маше» → contacts; несколько совпадений → голосовая дизамбигуация
- Confirm + revise-петля: `user.confirm.request` → показать текст → approve / `revision` («перепиши короче») → перегенерация → новый confirm
- Cadence guard: rate-limit, джиттер, запрет веера, запрет burst-серий (§14)
- Idempotency: `outbound_messages.idempotency_key`

**Тест-критерий:**
> UC-2: «ответь Маше, что буду в 7» → текст показан до отправки → после approve → отправлено через userbot → запись в outbound_messages.
> UC-10 (требует M5 + M2): найти слот в календаре + отправить предложение с revise-петлёй.
> Дубль-ретрай не шлёт сообщение дважды (idempotency_key).

## M7 — Заказ еды ⬜ не начато

**Что входит:**
- Browser-автоматизация заказа (hak-browser / UI-актуаторы, §6): собрать корзину, чекаут
- Spend cap + allowlist заведений/блюд (§14)
- Confirm выше порога (настраивается юзером)
- Идемпотентность: `orders.idempotency_key`
- Правило карты: агент пользуется привязанной картой, никогда не вводит/не хранит/не редактирует карточные данные (§0, принцип 5)
- Чекаут с 3DS/SCA — подтверждает пользователь сам

**Тест-критерий:**
> UC-5: «закажи как обычно» → корзина собрана → confirm → заказ размещён через браузер/UI.
> Дубль при retry → один заказ (idempotency_key).
> Превышение порога spend cap → стоп (kill_switch).

## M8 — Задачи и нарративность ⬜ не начато

**Что входит:**
- Полная модель задач (§20): `tasks` (goal, status, steps_done/steps_total, result_summary)
- Голосовое управление задачами: «стоп» / «отмени» / «что делаешь» / «продолжи» / «потом доделай»
- Haiku различает «заткнись» (рубит TTS) и «отмени» (рубит задачу)
- Отмена: cancel-флаг перед каждым шагом, ≤1 шага латентности
- Нарративность: анонс для задач >5 с, вехи по этапам скилла, краткий отчёт по завершении
- `task.status` стримится на клиент (renderer показывает прогресс)
- Presence: юзер ушёл → отчёт пушем на телефон, не голосом в пустую комнату
- Resume: продолжение с текущего шага (идемпотентность через verify)
- Ошибки: голосом причина + одно следующее действие; детали → `ui.display`-карточка

**Тест-критерий:**
> UC-11: Excel 40 шагов → «стоп» → задача paused, голосом «успел A и B — докончить или отменить?» → «продолжи» → resume без повтора сделанного.
> Актуатор останавливается ≤1 шага от «стоп».
> Юзер ушёл → пуш на телефон по завершении.

## Параллельный трек: mobile-компаньон (Android-first)

Нужен к M5. Независим от M1–M4.

**Что входит:**
- Android приложение (Kotlin / Java): сенсор гео + приёмник пушей
- Android Geofencing API: геофенсы вокруг home/gym/work → события на сервер по HTTPS
- FCM data-сообщения: получение `ProactiveNudge` с `expiresAt`; валидация на устройстве; просроченный — молча в лог
- `ACCESS_BACKGROUND_LOCATION`: для личной сборки/sideload — ок; публикация в Play потребует обоснования (§18)
- iOS: ambient-голос в фоне невозможен; компаньон на iOS = пуши + гео + foreground-голос (backlog)

**Статус:** 🟡 скелет (интерфейс готов, интеграция Firebase требует ключей)
