# Джарвис → идеал (как в фильмах + идеи Антона): готовность и дорожная карта

> Синтез по итогам 3 глубоких аудитов 2026-06-20 (концепт+надёжность 70, «думать» 33, универсальность+мультитенант 92 = 195 находок), живых прогонов и идей Антона (память project_jarvis_concept/architecture/...).
> Оценки готовности — честные, по фактическому состоянию кода (не по намерениям).

## 0. Что значит «идеал» (цель)
**Кино-Джарвис (Старк):** говоришь обычными словами — он понимает намерение и ДЕЛАЕТ на компе (и шире), мгновенно и в разговоре (перебиваешь — слушает), ВИДИТ экран и контекст, ПОМНИТ всё и учится, ПРЕДУГАДЫВАЕТ, с характером дворецкого, 100% попадание.
**Идеи Антона (память):** НЕ чат-бот (голос — основной канал, текст не главный), невидимо действует в ТВОИХ залогиненных сессиях, учится показом («сделай скилл»), помнит лучший способ → быстрее, универсально (инсталлер, юзеры/подписки), меню Навыки/Оплата/Язык/Контекст.

## 1. Готовность сейчас — честно (~55–60% к идеалу)
Сильное ядро (мозг+инструменты ~75%), но «ощущение Джарвиса» (мгновенный голос, 100%-надёжное управление, проактивность, продукт) — это и есть гэп.

|---|---|---|

## 2. Четыре столпа (примиряют гибкость+надёжность+скорость — фундамент всего)
1. **Verify-loop** — общая обёртка: действие→наблюдай исход→«достигнуто?»→повтор(bounded)→честно. Надёжность.
2. **Зрение-подложка** — screen_capture в петле где нет a11y (игры/canvas/нативные). Универсальность.
3. **Опытная память** — лучший верифицированный способ→fast-replay→self-heal. Скорость.
4. **Фоновая оптимизация** — на простое перепробовать альтернативы, апгрейдить плейбук.

## 3. Дорожная карта (фазы → идеал)

### Фаза 1 — «НАДЁЖНЫЕ РУКИ»: 100% попадание (самый большой рычаг ощущения «оно реально делает»)
- Общий **verify-loop как контракт** в agent-loop/skill-runner (закрывает 31 «assumed-ok»): меняющее-состояние действие обязано наблюдать исход (дешёвый сигнал→browser_read/inspect→screen_capture) + bounded-retry.
- **Инструменты наблюдения** (питают loop): `system_volume get`+readback, media-playback-state, post-write verify office, baseline-screenshot diff.
- **Зрение в петле** + `expect type='visual'` в skill-runner (для canvas/игр/видео) + авто-подсказка screen_capture на промахе a11y (сделано: `visionFallbackHint`).
- *Готово после фазы:* «сделай X» → реально сделано или честный провал, на ЛЮБОЙ поверхности.

### Фаза 2 — «МГНОВЕННЫЙ ГОЛОС» (ощущение живого собеседника)
> **СТАТУС 2026-06-21 (реальный тест):** Persistent WS был уже реализован + включён (флаг=1), но
> `deepgram.integration.test.ts` (живой TTS→ffmpeg→Deepgram) ДОКАЗАЛ баг: общий таймлайн сокета →
> хвост turn1 протекает в turn2 («раз 2 3» из прошлой команды). Per-utterance изолирован. Флаг
> ВЫКЛЮЧЕН (корректность > экономия churn). Тест — гейт для будущего фикса persistent (включить флаг →
> тест изоляции должен остаться зелёным). Попутно тест поймал **tail-loss**: агрессивный finalize
> (`close` ждёт 200мс при haveText) теряет ПОСЛЕДНЕЕ слово фразы («погода» отвалилась) — отдельный
> quality-баг тайминга (есть и в per-utterance), фикс = тюнинг wait vs латентность на реальных сэмплах.
- **Persistent Deepgram WS** (боль №1: open/close каждый ход) + KeepAlive/Finalize. ⚠️ нужен фикс кросс-ход изоляции (см. статус выше) ДО включения.
- Снизить латентность тяжёлого хода: филлер-ack как сейчас + потоковый вывод; рассмотреть каскад/realtime-архитектуру (LiveKit/Pipecat, Smart Turn).
- Акустический wake (или Deepgram keyterm) вместо текстового.
- *Готово после фазы:* отвечает за ~1с на простое, перебиваешь — слышит, разговор не рвётся.

### Фаза 3 — «УЧИТСЯ И УСКОРЯЕТСЯ»
- Обобщить **опытный плейбук** (пилот — резолв получателей) на ЛЮБУЮ задачу: {цель, контекст, лучший верифиц. способ, цена} → recall→fast-replay→verify→self-heal; хранить ЛУЧШИЙ (метрика шаги/латентность/надёжность; API>a11y>координаты).
- HERMES: семантический recall (vector-колонка) вместо лексики; обучение показом до конца.
- Фоновая оптимизация частых задач на простое.

### Фаза 4 — «ПРЕДУГАДЫВАЕТ» (проактивность кино-Джарвиса)
> **СТАТУС 2026-06-21:** salience/NudgeQueue/presence/scheduler-математика были построены+протестированы,
> но НЕ соединены (никто не звал `shouldInterrupt`). Построен **`proactive/hub.ts ProactiveHub`** (+7 тестов):
> уважительная доставка — `emit(trigger,ctx)` → salience-гейт → speak ИЛИ отложить (NudgeQueue) → `drain`
> при освобождении контекста → протухшее дроп; DND/locked/fullscreen уважаются, критический пробивает
> fullscreen. Инфраструктура проактивности теперь ПОЛНАЯ и протестирована.
> **ОСТАЛОСЬ (value-add, гейт = интеграции):** реальные ИСТОЧНИКИ событий (почта/календарь/цены/геофенс
> — нужны интеграции) + обучение привычкам (схема habits есть, не наполняется) + per-user routing речи в
> hub. Спекулятивные триггеры НЕ добавлять — это ровно over-triggering, на который жаловался Антон;
> любой источник делать КОНСЕРВАТИВНО (высокий порог важности, без флуда).
- Подключить стабы: salience ✅ (через hub), presence ✅, triggers (источники — TODO, нужны интеграции), habits (схема есть).

### Фаза 5 — «ДЕЙСТВУЕТ ГДЕ УГОДНО» (концепт без хардкода)
> **СТАТУС 2026-06-21:** общий `browser_act` кликал по `text.includes(query)` — короткий запрос давал
> ОПАСНЫЕ ложные клики: «да»→«Удалить» (`удалить`.includes(`да`)). Построен `@jarvis/shared/ui-match.ts
> bestTextMatch` (+7 тестов): точное>целое-слово>префикс>подстрока, короткий запрос (≤3) — БЕЗ подстроки
> → «да» больше не цепляет «Удалить». Это робастная grounding-основа общего пути.
> **ОСТАЛОСЬ (нужна ЖИВАЯ валидация — модель+браузер, риск регресса рабочих флоу):** вплести bestTextMatch
> в eval-строки browser_act (jarvis-browser/background.js) с проверкой на живом браузере; свернуть per-site
> Telegram/Я.Музыка-селекторы в gather→инспектор/зрение→bestTextMatch→клик→сверка (как сделано для
> резолва контактов); мессенджеры WA/Discord через общий web-путь + верификация доставки. Делать ПО ОДНОМУ
> сайту с live-тестом, рабочий путь держать фолбэком — НЕ слепой свап.
- ✅ Робастный матчер клика (ui-match); ⏭️ вплетение + per-site unwind — под live-валидацию.

### Фаза 6 — «ПРОДУКТ» (идеи Антона: раздать инсталлер)
- **Универсализация (A):** .env-поиск в Program Files, резолв Chrome/сайдкара, bind 127.0.0.1, авто-миграции, single-instance (см. UNIVERSALITY_MULTITENANT_PLAN.md).
- **Мультитенант (B):** auth (снять хардкод userId→validateToken→lazy-provision), per-user шифр. ключи (user_credentials), партиция сторов по userId, подписки (usage_quota→«Оплата»). Схема готова.
- **UI-концепт:** голос-first (текст не главный), орб состояния, меню Навыки/Оплата/Язык/Контекст, first-run онбординг (логины/ключи).

## 4. Честные блокеры (не всё в нашей власти)
- **Realtime-латентность Opus** на тяжёлом ходе — архитектурно (филлер маскирует, но «800мс на тяжёлом» не берётся). Кино-мгновенность ограничена моделью.
- **Telegram api_id** для настоящего голосового/MTProto — РФ-IP режет (см. handoff); поэтому невидимый CDP-webK, а не userbot.
- **Невидимая работа в реальных логинах** на Windows — webK рендерит только в видимом/occlusion-off окне (решено off-screen CDP, но это инфра-тонкость).
- **Сайдкар/Office/UWP** — детект + честная деградация, не «всё везде из коробки».

## 5. Рекомендуемый порядок
Фаза 1 (надёжные руки — максимум ощущения «работает») → Фаза 2 (мгновенный голос) → Фаза 6A (инсталлер, чтоб раздавать) → Фаза 3 (учится) → Фаза 6B (мультитенант/подписки) → Фаза 4 (предугадывает) → Фаза 5 (везде без хардкода).
