# CONTRIBUTING — Как работать с кодовой базой

---

## Принцип вертикальных срезов (§17)

**Не поднимать всё сразу.** Каждый milestone — полностью рабочий и тестируемый срез.
Перед тем как начинать M2, убедитесь, что M0 проходит тест-критерий из [docs/BUILD_PLAN.md](docs/BUILD_PLAN.md).

При добавлении новой фичи:
1. Определить, к какому milestone она относится.
2. Если milestone ещё не начат — оформить стаб с интерфейсом и пометкой `// TODO(Mx): ...`.
3. Стаб должен **типизироваться и не падать на импорте**. Это важнее, чем пустой `{}`.
4. Заполнять стаб — только в рамках работы над соответствующим milestone'ом.

## Грундинг по ролям, не по координатам (§0, принцип 2; §6)

**Никогда не хранить пиксельные координаты или CSS-селекторы в SKILL.md или в ActionCommand.**

Правильно:
```ts
{ kind: "ui.ground", query: { role: "button", name: "Отправить" } }
{ kind: "ui.invoke", target: { by: "handle", handle }, pattern: "invoke" }
```

Неправильно:
```ts
{ kind: "input.click", target: { by: "coords", x: 347, y: 892 } } // только как vision-fallback
```

Пиксельные координаты (`by:"coords"`) — исключительно vision-fallback, когда a11y-дерево пусто (видео-канвас TikTok/YouTube).

Скилл хранит **намерение и процедуру в терминах ролей/интентов** — это делает его устойчивым к редизайну интерфейса.

## Как добавить скилл (SKILL.md §8)

Скилл — файл формата `SKILL.md` для конкретного пользователя, хранится в таблице `skills.content_md`.

### Структура SKILL.md

```markdown
---
name: имя_скилла_snake_case
description: Что делает скилл (для роутера)
triggers: ["фраза активации 1", "фраза активации 2"]
tools: ["app.focus", "ui.ground", "input.type", "message.send"]
version: 1
surface: vk-desktop        # vk-desktop | youtube-web | ...
grounding: a11y             # a11y | vision | hybrid
---

## Шаги (в терминах ролей/интентов, НЕ координат)
1. app.focus → "ИмяПриложения"
2. ui.ground → role="list", name~="Имя раздела"
3. ui.invoke → найденный handle, pattern="invoke"
4. ui.ground → role="textbox", name~="Поле ввода"
5. input.type → текст
6. verify → постусловие всей задачи (файл существует, сообщение в «Отправленных»)

## Грабли
- Описать известные нестабильные места и как их обходить.
```

### Правила

- **Verify-шаг обязателен**: без него «успех» скилла — самообман. (§8)
- **Guard-шаги** (`message.send`, `code.run`, powershell) требуют `user.confirm.request`.
- Свежий скилл с guard-шагами из демонстрации — **обязательное ревью** перед первым применением. (§14)
- Поле `steps` в БД — derived-парс из `content_md`; редактировать только `content_md`. (§8)
- Не добавлять шаги без `expect`-постусловия там, где постусловие выразимо.

## Как устроены тиры (§7)

| Тир | Когда использовать |
|---|---|
| Tier 0 (локально, $0) | Детерминированные команды: `app.launch`, `app.focus`, regex-намерения |
| Tier 0.5 (skill-runner, $0) | Исполнение известного SKILL.md без LLM; эскалация только на fail или needsLlm-шаге |
| Tier 1 (Haiku) | Классификация намерений, salience, простые ответы, умные напоминания |
| Tier 2 (Sonnet) | Сочинение текста, работа по скиллу, анализ, синтез веб-результатов |
| Tier 3 (Fable) | **Разовое** освоение нового GUI, vision-планирование — дорого, один раз |

Каскад: пытаться снизу вверх. **Для голоса** тир выбирается Haiku-классификатором до генерации, а не retry-ом.

Цель: цикл шортсов «дальше» → skill-runner → $0. Нет скилла → Fable-сессия (один раз) → запись SKILL.md → следующий раз tier-0.5.

## Соглашения по коду

### Язык

- Код и комментарии — **на русском** (как в спецификации). Имена переменных/функций — английские идиоматичные TS.
- Ссылки на разделы спецификации в комментариях: `// §6 — грундинг по a11y`.

### TypeScript

- `"type": "module"` во всех `package.json` (ESM).
- `strict: true`, `noUncheckedIndexedAccess: true` — нарушений быть не должно.
- Импорты только через bare-спецификаторы: `"@jarvis/protocol"`, `"@jarvis/shared"` (не deep paths).

### Стабы

```ts
// TODO(M3): реализовать полный UIA-грундинг через sidecar-win (§6)
export async function groundElement(query: UiaQuery): Promise<Result<UiaHandle, string>> {
  return err("not_implemented");
}
```

Стаб должен:
1. Иметь правильный тип возвращаемого значения.
2. Не паниковать / не бросать необработанное исключение.
3. Содержать `// TODO(Mx): ...` с номером milestone.

### Result<T,E>

Использовать `ok(value)` / `err(error)` из `@jarvis/shared` для fallible-функций. Не бросать исключения там, где ожидаемая ошибка — часть протокола.

### Logger

```ts
import { createLogger } from "@jarvis/shared";
const log = createLogger("gateway");
log.info("соединение", { sessionId });
```

## Безопасность при разработке (§14)

- **Никогда не коммитить `.env`** с реальными ключами. Только `.env.example`.
- **Не добавлять карточные данные** (даже демо-номера) ни в какой файл репозитория. (§0, принцип 5)
- **Confirm-гард нельзя обойти** в тестах, упростив `actuators/`. В e2e-тестах — мокировать `ConfirmResult`, а не удалять guard.
- **powershell через code.run** — всегда `user.confirm.request{kind:"irreversible"}`. Не добавлять исключения.
- Перед PR с изменением guard-шагов — явно описать в описании PR, почему это безопасно.

## Структура package.json нового пакета/приложения

```json
{
  "name": "@jarvis/имя",
  "version": "0.1.0",
  "private": true,
  "type": "module",
  "dependencies": {
    "@jarvis/protocol": "workspace:*",
    "@jarvis/shared": "workspace:*"
  }
}
```

`tsconfig.json` нового пакета:
```json
{
  "extends": "../../tsconfig.base.json",
  "include": ["src/**/*"]
}
```

(Путь к `tsconfig.base.json` — относительный от расположения файла.)

## Pull request checklist

- [ ] Тест-критерий соответствующего milestone'а проходит (или явно отмечено, что milestone ещё не закрыт).
- [ ] Стабы типизируются и не падают на импорте.
- [ ] Нет прямых путей к node_modules (только workspace-импорты).
- [ ] Комментарии на русском, ссылки на §N где уместно.
- [ ] Карточные данные не затронуты ни в одном файле.
- [ ] Guard-шаги не обойдены.
- [ ] `pnpm -r build` проходит без ошибок.
